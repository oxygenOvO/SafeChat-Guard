# SafeChat-Guard 项目阶段汇报

> **项目名称**：SafeChat-Guard —— 面向中文对话场景的大模型输入/输出违规内容过滤与价值观安全评估系统
> **对应赛题**：题目 1 · 面向对话场景的大模型输入/输出违规内容过滤系统
> **汇报日期**：2026 年 7 月 7 日

---

## 一、项目概述

大语言模型在聊天、问答、客服等场景广泛使用，用户输入或模型输出常出现低俗、暴力、广告、谣言等违规内容，带来使用风险。本项目依据赛题要求，基于开源大模型接口，搭建一套轻量化内容过滤系统，实现**输入拦截、输出修正、日志记录**等基础防护能力，并在赛题基础要求之上引入**中文对抗归一化**与**价值观五维安全评估**两项创新特色。

系统已实现端到端可运行：用户在 Web 界面输入一句话后，系统依次完成归一化、双层过滤、分级处理、模型调用、输出校验与日志记录，并对外提供统计接口。

## 二、系统总体架构

### 2.1 技术路线

系统采用分层流水线架构，处理流程如下：

```
用户输入
   │
   ▼
① 归一化层（normalizer）  —— 全半角/大小写/emoji/噪声/重复/变体/谐音/拼音/缩写
   │
   ▼
② 第一层过滤（rule_filter）—— 关键词词库 + 正则规则快速匹配
   │   未命中 ↓
   ▼
③ 第二层过滤（semantic_classifier）—— 语义分类模型二次判断
   │
   ▼
④ 分级处理（pipeline）    —— 高风险拦截 / 低风险脱敏 / 正常放行
   │
   ▼
⑤ 调用大模型（llm_client）—— 安全输入转发至轻量大模型
   │
   ▼
⑥ 输出二次校验（rule_filter 复用）—— 对模型回复再次检测
   │
   ▼
⑦ 日志与统计（logger）    —— 记录输入/输出/命中规则/处理结果/时间
```

### 2.2 模块划分

| 模块 | 文件 | 职责 |
| --- | --- | --- |
| 归一化子系统 | `safechat_guard/normalization/` | 中文对抗文本归一化，含 8 类归一化器 |
| 规则过滤 | `safechat_guard/rule_filter.py` | 词库匹配 + 正则匹配，输出检测结果 |
| 语义分类 | `safechat_guard/semantic_classifier.py` | 第二层语义判断（当前为占位实现） |
| 脱敏器 | `safechat_guard/sanitizer.py` | 命中词替换为 `***` |
| 流水线 | `safechat_guard/pipeline.py` | 串联全流程，执行分级决策 |
| 大模型客户端 | `safechat_guard/llm_client.py` | 对接大模型 API（当前为 Mock） |
| 日志记录 | `safechat_guard/logger.py` | JSONL 事件日志读写 |
| Web 服务 | `app.py` | HTTP 服务，提供 `/api/chat` 与 `/api/stats` |
| 前端 | `templates/index.html`、`static/` | 聊天界面与结果展示 |
| 数据 | `data/lexicons/`、`data/rules/`、`data/maps/` | 词库、规则、归一化映射 |

## 三、已完成功能详解（对照赛题要求）

赛题要求实现四大基础功能与一项辅助功能，下文逐项对照说明完成情况。

### 3.1 违规词库与规则构建 ✅ 已完成

**赛题要求**：自行整理常见违规词汇、短句、正则规则，分为色情、暴力、广告、敏感话术四类；支持手动添加、删除、导入违规词。

**完成情况**：

- **四类词库**已整理完成，存放于 `data/lexicons/*.txt`，每行一个词，支持 `#` 注释：

| 词库文件 | 类别 | 词条数 |
| --- | --- | --- |
| `ad.txt` | 广告 | 123 |
| `porn.txt` | 色情 | 304 |
| `sensitive.txt` | 敏感话术 | 781 |
| `violence.txt` | 暴力 | 615 |
| **合计** | — | **1823** |

- **正则规则**存放于 `data/rules/regex_rules.json`，共 **7 条**，覆盖手机号、微信号/联系方式、邮箱、IPv4、HTTP(S) URL、www 网址、域名，每条规则含 `name / category / pattern / level / score / reason` 字段。
- 词库与规则均以**文件配置**形式组织，运行时由 `RuleFilter` 自动加载；新增/修改词库后只需编辑文件即可生效，具备手动添加、删除、导入的扩展基础。

> **待补**：目前词库管理以"编辑文件"方式完成，尚未提供 Web 端增删改查 API，详见第六节。

### 3.2 双层过滤逻辑 ✅ 第一层已完成，第二层为占位

**赛题要求**：第一层关键词 + 正则匹配，快速拦截明显违规内容；第二层调用开源预训练文本分类模型，对疑似内容做二次语义判断。

**第一层（关键词 + 正则）—— 已完成**

`RuleFilter.detect()` 实现两层匹配：
1. **词库匹配**：遍历四类词库，对文本做子串匹配。色情/暴力类命中记为 `high`（评分 80），广告/敏感类命中记为 `medium`（评分 55）。
2. **正则匹配**：遍历 7 条正则规则，忽略大小写匹配，按规则配置的 `score` 记分。

每个检测结果封装为 `Detection`（类别、等级、评分、原因、来源、命中词列表）。

**第二层（语义分类）—— 占位实现**

`SemanticClassifier` 当前为简单短语匹配占位，用于保证全流程可运行。计划在规则基线稳定后替换为轻量中文安全分类模型（如 XLM-R / RoBERTa 跨语言迁移方案），并利用大创的低资源数据增强思路构造 300–800 条中文样本。

> **待补**：替换为真实分类模型，详见第六节。

### 3.3 分级处理逻辑 ✅ 已完成

**赛题要求**：高违规内容直接拦截；低风险疑似内容自动脱敏替换违规词后再送入大模型；正常内容正常请求大模型并返回结果。

**完成情况**：

`SafeChatPipeline._filter_text()` 依据最高风险分执行三级决策，阈值由 `config.yaml` 配置：

| 风险分 | 处理动作 | 说明 |
| --- | --- | --- |
| ≥ 80 | `block` 拦截 | 拒绝向大模型转发，向用户提示"输入被拦截" |
| 40–79 | `sanitize` 脱敏 | 由 `Sanitizer` 将命中词替换为 `***`，再送入大模型 |
| < 40 | `pass` 放行 | 正常请求大模型 |

脱敏器按命中词长度从长到短排序后逐个替换，避免短词误伤长词，尽量保留原句语义结构。

### 3.4 模型输出二次校验 ✅ 已完成

**赛题要求**：对大模型返回的回答再次做内容检测，若存在违规内容，屏蔽并替换为标准合规话术。

**完成情况**：

`handle_chat()` 在拿到模型回复后，复用 `_filter_text()` 对输出再次检测：
- 输出命中高风险 → 替换为标准合规话术："The model output was blocked because it contains high-risk unsafe content."
- 输出命中中风险 → 脱敏后再返回
- 输出正常 → 直接返回

实现了输入侧、输出侧的**双向防护**。

### 3.5 日志与统计 ✅ 已完成

**赛题要求**：记录每条请求的检测结果、处理方式、时间；简单统计每日违规请求数量、违规类型占比。

**完成情况**：

- **日志记录**：`EventLogger` 以 JSONL 格式逐条追加写入 `data/logs/events.jsonl`，每条记录自动带 UTC 时间戳。一次完整对话记录包含：`stage / input / safe_input / raw_reply / final_reply / input_filter / output_filter`，覆盖输入、归一化后文本、原始回复、最终回复及两侧过滤结果。
- **统计接口**：`/api/stats`（GET）由 `pipeline.stats()` 计算，返回总事件数、拦截次数、脱敏次数。
- 当前日志已积累 **23 条**事件记录。

> **待补**：统计目前仅返回总计数，未按"每日/类型占比"维度细分，前端为 JSON 文本展示，详见第六节。

### 3.6 创新特色（赛题定位 SafeChat-Guard）

项目在赛题基础要求之上规划两项创新特色：

**特色一：中文对抗归一化 ✅ 已完成（核心亮点）**

针对中文场景下谐音字、形近字、拼音、缩写、emoji 等对抗变体，构建了完整的归一化子系统 `safechat_guard/normalization/`，包含 **8 类归一化器**：

| 序号 | 归一化器 | 作用 |
| --- | --- | --- |
| 1 | UnicodeNormalizer | 全角/半角字符统一 |
| 2 | CaseNormalizer | 大小写统一 |
| 3 | emoji 映射 | emoji 还原为语义文本 |
| 4 | NoiseCharNormalizer | 零宽、噪声字符清洗 |
| 5 | RepeatCharNormalizer | 重复字符压缩（如"啊啊啊"→"啊啊"） |
| 6 | 变体字映射 | 形近字/变体字还原 |
| 7 | 谐音映射 | 谐音字还原（如"微Ⓢì信"→"微信"） |
| 8 | 拼音/缩写映射 | 拼音、缩写还原 |

归一化映射数据存放于 `data/maps/*.json`（共 5 个映射文件）。子系统支持 `normalize_with_trace()` 追踪每一步变更，便于效果对比与可解释性展示。配套 **5 个单元测试全部通过**，覆盖 Unicode 宽度、噪声字符、emoji、拼音、trace 追踪等场景。

该特色有效提升了对抗变体写法的识别率，是与"未归一化基线"做效果对比的基础。

**特色二：价值观五维安全评估 🔲 规划中**

计划将大创项目的"价值观五维评估"轻量化为**和谐、友善、法治、诚信、文明** 5 个维度的安全评分面板，作为可视化亮点与创新点。目前尚未开始实现。

## 四、系统运行方式

### 4.1 运行环境

- Python 3.13，**纯标准库实现，零第三方依赖**（第一版）
- 启动命令：`python app.py`
- 访问地址：`http://127.0.0.1:8000`

### 4.2 对外接口

| 方法 | 路径 | 功能 |
| --- | --- | --- |
| GET | `/` | 聊天 Web 界面 |
| POST | `/api/chat` | 提交消息，返回过滤结果与模型回复 |
| GET | `/api/stats` | 返回统计（总事件数、拦截数、脱敏数） |

### 4.3 配置

`config.yaml` 集中管理应用、风险阈值、大模型、日志配置：

```yaml
risk:
  block_threshold: 80      # 拦截阈值
  sanitize_threshold: 40   # 脱敏阈值
llm:
  provider: mock           # 当前为 Mock，预留 Qwen 接入
  model: qwen-tiny
```

## 七、存在的问题与不足

1. **第二层语义分类为占位实现**：`SemanticClassifier` 仍是简单短语匹配，未接入真实中文安全分类模型，双层过滤的第二层尚未真正生效。
2. **大模型未接入**：`llm_client.py` 为 `MockLLMClient`，`config.yaml` 中 `provider: mock`，尚未对接 Qwen-Tiny / ChatGLM 等开源大模型 API。
3. **测试用例不足且存在编码损坏**：测试样例仅 4 条（3 条损坏），离赛题交付要求差距较大；归一化与规则层之外缺少独立测试。
4. **统计维度较粗**：`/api/stats` 仅返回总计数，未按"每日违规数量、违规类型占比"细分；前端统计为 JSON 文本展示，缺少图表可视化。
5. **词库管理缺 API**：目前词库维护依赖编辑文件，未提供 Web 端增删改查接口。
6. **创新特色二未启动**：价值观五维安全评分面板尚未开始实现。

## 九、小结

SafeChat-Guard 已完成**端到端可运行的过滤系统骨架**：四类词库（1823 词）与 7 条正则规则支撑的第一层过滤、三级分级处理（拦截/脱敏/放行）、模型输出二次校验、JSONL 日志与统计接口均已落地，Web 界面可正常对话与查看检测结果。

作为核心创新点，**中文对抗归一化子系统**（8 类归一化器 + trace 追踪）已完整实现并通过测试，有效提升了对谐音、拼音、emoji 等对抗变体的识别能力。

当前主要待推进工作集中在"智能层"（语义分类模型与大模型 API 接入）与"测评层"（测试用例扩充与过滤效果量化），这两项完成后即可全面满足赛题的功能完整性与过滤效果评价标准。
